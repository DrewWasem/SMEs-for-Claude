Experts.

A framework for packaging deep domain knowledge into shareable, on-demand modules for Claude Code. SMEs sit alongside your skills, agents, and MCP servers — dormant until called, gone when done.

The Concept

What exists today, what's missing, and what SMEs add
The Gap

Claude Code has skills (things it can do), agents (ways it can delegate), MCP servers (tools it can connect to), and memory (things it remembers). But it's missing domain expertise — the deep, structured knowledge that turns a generalist into a specialist. An SME fills that gap.

What Each Primitive Does
⚡
Skills
What Claude can do. Discrete actions like "commit", "build cache", "create plan".
🤖
Agents
How Claude delegates. Autonomous subprocesses that handle complex tasks.
🔌
MCP Servers
What Claude can connect to. External tools, APIs, and data sources.
🧠
SMEs
What Claude deeply knows. Domain expertise, judgment, constraints, and taste.
The Specialist Analogy

A skill is like knowing how to use a stethoscope. An agent is like having a medical assistant. An MCP server is like having access to a lab. But an SME is like having a cardiologist in the room — someone who knows when to use the stethoscope, what to listen for, what questions to ask, and what "normal" sounds like in this specific domain.

The Concrete Example
// Without an SME:
Skill  = "I can write narration text"
Agent  = "I can research, plan, then write it"
MCP    = "I can read the project files"

// With an SME:
SME    = "I know this is for kids 7-11, so the
          narration must be under 20 words, present
          tense, NEVER say 'Great job!', reference
          the game world not the child, and make
          funny fails funnier than success."

Architecture

How SMEs fit into the Claude Code ecosystem — without modifying it
Where SMEs Live

SMEs sit right next to your existing skills, agents, and memory — same level, same directory pattern, same discoverability. 

.claude/
├── skills/          # What Claude can DO
│   ├── commit/
│   ├── research/
│   └── sme/           # ← The SME runner skill
│
├── smes/            # What Claude deeply KNOWS
│   ├── child-game-design/
│   ├── 3d-game-dev/
│   └── postgres-dba/
│
├── agents/          # How Claude DELEGATES
├── memory/          # What Claude REMEMBERS
└── commands/        # User slash commands
The On-Demand Lifecycle

SMEs are completely dormant until invoked. They load into a subagent, do their work, return the result, and release all context. Zero residual overhead.

Non-Breaking Integration

SMEs use existing Claude Code extension points. They never modify anything outside their own directory.

Existing MechanismHow SMEs Use It
.claude/skills/SME runner lives here as a standard skill
Task toolSMEs spawn as subagents via the Task tool
Read/Glob/GrepSME subagents use standard tools to read project files
CLAUDE.mdUntouched. SMEs are independent.
.mcp.jsonUntouched. SMEs don't register MCP servers.

Anatomy of an SME

Every file has a purpose. Nothing is optional decoration.
Directory Structure
my-sme/
├── sme.yaml              # Manifest: identity, capabilities, tool access
├── system-prompt.md       # THE prompt that creates the domain expert
├── knowledge/             # Domain knowledge (tiered loading)
│   ├── constraints.md     # Hard rules — ALWAYS loaded (~50 lines)
│   ├── heuristics.md      # "When X, do Y" — ALWAYS loaded (~100 lines)
│   ├── ontology.md        # Domain model — ON DEMAND (~200 lines)
│   ├── taste.md           # Good/bad examples — ON DEMAND
│   └── glossary.md        # Vocabulary — ON DEMAND
├── skills/                # SME-scoped skills (only inside subagent)
└── examples/              # Few-shot demonstrations
sme.yaml — The Manifest

Identity, capabilities, dependencies, and loading configuration.

name: child-game-design
version: 2.0.0
description: >
  Expert in designing AI-powered
  games for children ages 7-11.

tools:
  - Read
  - Glob
  - Grep

capabilities:
  - "Review kid-facing text"
  - "Audit scene scripts"
  - "Design game tasks"

context:
  always:
    - knowledge/constraints.md
    - knowledge/heuristics.md
  on_demand:
    - knowledge/ontology.md
    - knowledge/taste.md
system-prompt.md — The Brain

This single file transforms a generalist Claude into YOUR domain expert. The most important file in any SME.

# Child Game Design SME

You are a Subject Matter Expert
in designing AI-powered games
for children ages 7-11.

## Your Expertise
- Piaget's Concrete Operational stage
- Age-appropriate UX patterns
- Comedy-first pedagogy
- Brand voice and visual identity

## Response Format
- Verdict: Pass / Needs Work / Fails
- Findings: Specific issues
- Recommendations: Concrete fixes

## You Are Not
- A general-purpose assistant
- A yes-man
- Abstract — always concrete

Knowledge Tiers

Not all knowledge is loaded at once. Tiers manage the token budget.
The Tiered Loading Model

The "always" tier is small and loads every time. The "on demand" tier is richer and loads only when the subagent determines it needs it. This keeps the base invocation fast and focused.

TierFileSizeWhenPurpose
Always constraints.md ~50 lines Every invocation Hard rules. Non-negotiable. "NEVER do X."
Always heuristics.md ~100 lines Every invocation Judgment patterns. "When X, usually do Y."
On Demand ontology.md ~200 lines When needed Full domain model. Concepts and relationships.
On Demand taste.md ~150 lines When needed Good/bad examples. Quality calibration.
On Demand glossary.md ~50 lines When needed Domain vocabulary and terminology.
Constraints

The smallest and most critical file. Lists things that must NEVER happen. Loaded every time because violating these is always wrong.

"NEVER use red for failure."
"NEVER say 'Great job!'"
Heuristics

Pattern-matched judgment. Not absolute rules, but strong defaults. "When you see this situation, usually do this."

"When narration exceeds 20 words, trim it."
"When a button is smaller than 48px, flag it."
Taste

The most powerful calibration tool. Good/bad example pairs. Claude learns more from 5 pairs than 50 rules.

GOOD: "The monster catches the cake!"
BAD: "Great job completing the task!"

Invocation

Three ways to call an SME — all producing the same result
# 1. User invokes directly
$ /sme child-game-design "Review this narration: 'Great job!'"

# 2. List available SMEs
$ /sme list
child-game-design Expert in kids 7-11 game UX
3d-game-development Browser 3D with Three.js/R3F
prompt-engineering Teaching prompt eng through play

# 3. Get SME details
$ /sme info child-game-design
Capabilities: review text, audit scenes, design tasks...
Knowledge: 5 files (constraints, heuristics, ontology...)
Tools: Read, Glob, Grep
Example: SME Review in Action
Input
/sme child-game-design "Review this narration: 'Wonderful work! You successfully completed the challenge by providing detailed prompts.'"
SME Response
Verdict: Fails Constraints
Findings: (1) "Wonderful work!" — banned empty praise. (2) "completed the challenge" — "challenge" is banned edu-jargon. (3) "providing detailed prompts" — "prompts" is banned; say "description". (4) 13 words about the kid, 0 about the game world.
Fix: "The monster catches the giant cake and does a happy dance! Confetti everywhere."
Claude Can Also Call SMEs Autonomously

Just like Claude decides when to use the Task tool or spawn an agent, it can decide when to invoke an SME. If it recognizes a task requires domain expertise it doesn't have loaded, it can spawn the appropriate SME subagent, get the expert review, and incorporate the findings — all without the user explicitly calling /sme.

Create Your Own

A template is included — copy it, fill it in, start using it
# Copy the template
$ cp -r template/ smes/my-domain/

# Edit the manifest
$ edit smes/my-domain/sme.yaml

# Write the brain
$ edit smes/my-domain/system-prompt.md

# Fill in knowledge
$ edit smes/my-domain/knowledge/constraints.md
$ edit smes/my-domain/knowledge/heuristics.md

# Install and use
$ cp -r smes/my-domain /project/.claude/smes/
$ /sme my-domain "review this thing"
Writing Tips
●Constraints should be tiny — ~50 lines. They load every time. Keep them tight and non-negotiable.
●Be concrete, not abstract — "NEVER use red for failure" beats "use appropriate colors."
●Taste examples are powerful — Claude calibrates more from 5 good/bad pairs than 50 rules.
●Test with edge cases — the best SMEs handle ambiguous situations because heuristics cover gray areas.
●System prompt is everything — spend 80% of your effort here. It's the file that creates the expert.
●Define what it's NOT — scope creep kills SMEs. A postgres SME shouldn't give React advice.

Sharing

Each SME is a self-contained directory — share one at a time
Share One SME
# Zip one SME and send it
$ zip -r child-game-design.zip smes/child-game-design/

# Recipient installs it
$ unzip child-game-design.zip -d .claude/smes/

# Or just copy the directory
$ cp -r smes/child-game-design /other/project/.claude/smes/

# Or git clone just one SME (sparse checkout)
$ git clone --filter=blob:none --sparse <repo>
$ git sparse-checkout set smes/child-game-design

Design Principles

The rules that keep SMEs clean, composable, and non-breaking
😴
Dormant Until Called
Zero context cost when not in use. No tokens consumed, no processing overhead, no interference with normal operation. An SME that isn't invoked is invisible.
🛡
Subagent Isolation
Each invocation spawns a fresh subagent. No cross-contamination between SMEs, no residual context in the parent conversation. The specialist comes in, does the work, leaves.
🔒
Non-Breaking
SMEs use existing Claude Code extension points (skills, Task tool, standard file tools). They NEVER modify CLAUDE.md, settings.json, .mcp.json, or any other configuration. Install = copy. Uninstall = delete.
📦
Self-Contained
Every SME is one directory with everything it needs. No external dependencies beyond standard Claude Code tools. No build step, no compilation, no registration ceremony.
🔃
Independently Shareable
Share one SME at a time. Each is a self-contained directory. Zip it, git it, copy it. The recipient drops it into .claude/smes/ and it works.
📈
Composable
Multiple SMEs coexist without conflict. Call them independently or chain them sequentially. A task that needs both "3D game dev" and "child game design" expertise can consult both — each in its own subagent.
